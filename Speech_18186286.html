Tianxi Chen Background & Project Proposal

Now we’d like you to introduce the context of out project, which is the FarmBot Web Application. FarmBot is a device that is available for the general public to purchase. It’s aim is to create a scientific observation, such that the user is able to grow their vegetables in their backyards with assist from the information collected. Without being specifically expertise in agricultural field.

 

Here is a sample image that demonstrates the process. The FarmBot device can be controlled by the user via the Web Application. The App will provide user perceptions from the device, including camera, the position of the device, the conditions of the plants, whether the user should water the garden based on weather conditions etc., and execute a level of automated steps such as water the garden when predetermined by user.

The FarmBot demo interface is basically enabling user to access web app with a trial account, that enables the user to experience the features that they could have after own and maintain an actual FarmBot device. 

 

Currently, there’s a fair level of feature that user can experience such as design the garden. However, when under demo interface, users will receive message like this most of the times. Which is quite disruptive to the user experience.

Xinxin Guo Challenges + overcome

Here are challenges when we build up our features and how we overcome them.

Firstly, communication, especially internal communication, the challenge lies in not just speaking but in actively listening and understanding each other. We hold regular meetings to ensure face to face communication and document every decisions and progresses to ensure everyone up to date. The common issue among our project is clients may not fully grasp the project details, and provide feedback in time. To have better communication with client, we send a copy of summarized project proposal and ask question for details in slack.

The existed codebase have more than a thousand files included and 12k commitments, with this find ourselves tasked in such a exploit codebases can be super complex. This are all farmbot files, and u can see it is time consuming to find the one we need. We need to read through the develop documentation and fully understand the original code before making any changes. To help team members understanding the codebase and their task, we hold meeting for code inspection and included a communication diagram to visualizing communication, each group member is responsible for a section, can solve related issues involving in that section. As well we build detailed github branching standard, making sure every changes are clear and understandable.

Moreover, when writing automated test code, we need to recreate the environment that the program encounters during actual operation. This often involves handling many props and states, with the code frequently scattered across various files. Moreover, these files might contain additional related functions, even small changes may lead to unpredictable serious side effects when employing mocking features of testing frameworks like Jest.We inspected the domain and codebase together to find solution, and leverage various auxiliary tools in IDE.

Lastly, we also met obstacles while we setting up virtual environment using GCP. Initially, we only created a single virtual machine (VM) to host the Farmbot application as the VM was configured to run the necessary software. However, we encountered performance and connectivity issues, like slow access and frequent disconnections when attempting parallel access. In this case, we decided to create separate VM for each group members to improve performance especially when dealing with multiple concurrent connections. Even with the right VM seting, we had to reset new private and public keys, set up new GCP instances, and redo the deployment process multiple times to accommodate this as there were still connectivity issues.

Mingyu Li Lessons

Lessons!! So, the most valuable lesson we learnt is how to work as a team in an agile way. And the first thing we need is to communicate effectively.

“Effective communication” here means it should be clear and precise about what progress has been made, what decisions are taken, what goals are set, and after communication, all team members are clear about the current progress of the project and know their respective responsibilities.  

By doing so, we can ensure that everyone is on the same page and working towards one common goal, which can significantly improve the collaboration and problem solving.

But communication only is apparently not enough because there might be information loss and different people can interpret the information differently during communication, which means that inconsistency, confusion is produced. That is why we need documentation.

A professional documentation should accurately record all relevant information about the current progress and should be updated frequently to ensure the consistency. And a standard to check if a documentation is good or not is whether it can be understood by someone new to the project.

So, A good documentation ensures that all team members share a common understanding of the project’s components, requirements, and goals, no confusion is caused, and save a lot of energy from communication.

And, with the communication and documentation, we can come up with a model for our agile development. It is quite similar to the observer pattern. The subject here is common understanding of the project with each member of the team sticking to it. Once someone made a progress, they will update their progress through communication and record it in documentation. And everyone else update their knowledge and adapt if necessary.

With this model, people can understand how the agile development work and how to join it.

 

Another lesson we learnt is how to modify and extend functionalities on an existing open-source project.

So, to extend features, the first thing is to understand the application. Our FarmBot Web App is a Redux React application with TypeScript. In such an application, there is a single source of truth called Local store, and we store the states of all components in it. So, if we need to update data, there is single place we have to update. And all components of UI will be immediately notified to reflect the changes.

However, to realise the demo account features, we cannot modify the current system directly because it is a bad practice and will cause some security concerns. So, to avoid it, we simulate a local store and let all components listen to it if it is in demo mode. With this method, we successfully implement all features without influencing the original functionalities.
