Decision making

Component tree rather than Domain model:

Our project primarily focuses on the development of the current UI, which is a a TypeScript and React-based system built by React components (Each file corresponds to a component, to an element in UI). So a component tree with hierarchy can help us better understand the structure of UI than a domain model with only entities and relationships but no hierarchy.

Sequence diagram:

Sequence diagram can visually show the interactions between potential users and the FarmBot system as well as the interactions inside the system, which allows developers and stakeholders to understand how a specific task is accomplished by different elements within the system.

Component diagram:

Component diagram represents how different parts of FarmBot system are organised together, providing a high-level view of the system's architecture.

Document changes:

Since we are introducing new features into the existing application, it is important for us to track which files we will modify and which components we will add so that we can understand how newly built features work and how they integrate into the current system.

Database model:

Database model is not relevant since we don’t use database in this project.

Deployment diagram:

We are not using CICD so we do not have an auto deployment workflow. All the deployment are done manually. Therefore, a deployment diagram is not relevant.

Coding standard:

In order for us to understand each other’s code and collaborate with each other more easily, we determine the coding standards to follow.
