Task allocations:

Since this is a project that is most relative to front-end development (simulating a FarmBot), we have decided to organise the team structure as 3 front-end developer and one backend developer to support this project.

Work environment:

We have decided to host one virtual machine for everyone to join. The benefit of this setting is we can possibly run multiple servers of the web app locally in the VM, so everyone should have access development process under the same environment

UI design:

As of now, the features we intend to add will have no impact on the current layout of the web app's UI. So, the UI design is not relevant at the moment.

That is, for each of epic 1 and 2, camera and movement, we have allocated two group members to maintain and develop the functions. This is the decision due to scale of each task to be achieved. Also, one person to manage the webcam feature.

8.25

Work environment:

Unlike before, we have decided to host virtual machine individually for each person. This is due to our initial environement failed when we attempt to host multiple farmbot web apps in parallel, and the host crashed. Under this new development set up, we are able to develop under same environment, without impacting other changes currently under progress.

User stories:

We have decided that these three Epics that we currently have are Must Have features. Since in a demo web app, we want to attract the potential users to try the core or essentials features that can dramatically improve their life by purchasing the devices.

We will add more user stories to ensure that the overall scope is suitable for the duration of entire development phase, and also for the client’s interest.

8.27

Add communication diagram:

Despite having various tools at our disposal to aid in teamwork and the agile process, we still experience some inefficiencies in collaboration. To resolve this issue, our team decides to use RACI matrix to document roles each person will play in different tasks. With this method, everyone involved can understand their responsibilities and the responsibilities of others and know the person to turn to when encountering problems, which prevent confusion and greatly promote collaboration.

 

Add new features:

  • Tour Guide feature: we find that the current tour guide lacks sufficient clarity, so we decide to modify it to make it more clear. This is a should have requirement since a good tour guide can improve users experience as well as maximise the value that users can derive from the app, increasing the likelihood of them to buy a product.

  • Support multiple models: we decide to make the demo support multiple models so that user can compare characteristics and limitations of them. This is a could have feature since it might be technically complex.

  • Advertisement: System can pop up farmbot ads to users. However, this feature might disrupt the user's browsing, making them unsatisfied. So we label this as could have.

  • Feedback System: Client can see feedback from the web app. This is a won’t have requirement with the lowest priority since it is not essential to achieve the core objective of the project.

 

8.29

Component tree rather than Domain model:

Our project primarily focuses on the development of the current UI, which is a a TypeScript and React-based system built by React components (Each file corresponds to a component, to an element in UI). So a component tree with hierarchy can help us better understand the structure of UI than a domain model with only entities and relationships but no hierarchy.

Sequence diagram:

Sequence diagram can visually show the interactions between potential users and the FarmBot system as well as the interactions inside the system, which allows developers and stakeholders to understand how a specific task is accomplished by different elements within the system.

Component diagram:

Component diagram represents how different parts of FarmBot system are organised together, providing a high-level view of the system's architecture.

Document changes:

Since we are introducing new features into the existing application, it is important for us to track which files we will modify and which components we will add so that we can understand how newly built features work and how they integrate into the current system.

Database model:

Database model is not relevant since we don’t use database in this project.

Deployment diagram:

We are not using CICD so we do not have an auto deployment workflow. All the deployment are done manually. Therefore, a deployment diagram is not relevant.

Coding standard:

In order for us to understand each other’s code and collaborate with each other more easily, we determine the coding standards to follow.

9.20

Deployment & Handover:

Deployment:

unlike in a traditional project, we’re contributing and developing new features and functions towards an open source project. Hence we should follow the provided development standards by Farmbot official documents and github page.

Therefore, we have decided to deploy our service using strategy of IaaS using Goggle Compute Engine. This effectively starts a virtual machine instance running the ubuntu system, which is the correct method as acknowledged by the official Farmbot Web App development github.

With this deployment strategy, we can develop function that can be easily adapted towards the original development file.

Handover:

For our scope of project, our ultimate handover would just be a pull request from the staging branch towards the original github of Farmbot Web App.

To manage our handover, we will develop a table of contents, including all relevant files and artifects, a changelog to record the functions that has been added to the existing system. Also, a readme file would also be essential for the client’s benefit in understanding.

Details of handover will be confirmed soon as we approaching the handover stage.

9.22

Front end design:

In the previous phase, we initially considered frontend design as not relevant. However, as the project progressed and based on client feedback, we have come to the realization that incorporating advertising and a tour guide in the Farmbot demo is indeed essential. Consequently, we have made the decision to undertake frontend design for these two new features.

A well-designed frontend enhances the overall user experience, making it easier and more enjoyable for users to interact with your application or website. This leads to better usability, reducing the learning curve for users and increasing their efficiency when using your product.
